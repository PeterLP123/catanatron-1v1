# GPU experiment backlog

This is the executable, evidence-gated queue for the UCL GPU workflow. No queue experiment
has completed as of 2026-07-12. Historical evaluation numbers are provisional, so complete
the corrected legacy rebaseline in [RESULTS_LOG.md](RESULTS_LOG.md) before interpreting a
new training result.

Definitions and predicates live in `catanatron.gym.experiment_backlog`. A run directory or a
manifest phase of `done` is not enough to unlock the next job: the backlog inspects smoke
artifacts, report integrity/signatures, weak gates, paired reward evidence, BC regret, and
promotion strength. Completed experiments resolve to `accepted`, `rejected`, or
`inconclusive`; pending work may be `ready`, `blocked`, or `skipped`.

## Use the queue

```bash
cd ~/catanatron-1v1
source "$HOME/.venvs/catanatron-1v1/bin/activate"

python examples/colonist_1v1_backlog.py list
python examples/colonist_1v1_backlog.py next
python examples/colonist_1v1_backlog.py show 05-mcts-strength-sweep
```

Launch the first GPU smoke in the visual command center:

```bash
python examples/colonist_1v1_backlog.py start 00-gpu-smoke
```

The launcher refuses blocked/skipped jobs and validates required inputs. `--force` is only
for evidence that exists outside the configured `runs/` root; it does not turn missing
evidence into a valid result.

## Validation stage: smoke and MCTS strength

The smoke gate requires the 20k run to finish with expected checkpoints and mid-run
evaluation and without a CUDA error. After it is accepted, run the CPU search sweep:

```bash
python examples/colonist_1v1_backlog.py start 05-mcts-strength-sweep
```

The command executes the fixed 10/25/50/100 ms, `F`/`AB:2`, two-seat, held-out-seed matrix
and writes `runs/05-mcts-strength-sweep/mcts_strength_sweep.json`. It must include p95
latency and every required cell. The report is atomically updated after each completed cell;
re-running the same command resumes compatible partial evidence and refuses a different
commit or protocol. Monitor completed cells, elapsed time, ETA, and the active cell with:

```bash
watch -n 5 python examples/colonist_1v1_search_benchmark.py \
  --report runs/05-mcts-strength-sweep/mcts_strength_sweep.json \
  --status
```

The cheaper command below is useful for profiling but is explicitly insufficient for the
evidence gate:

```bash
python examples/colonist_1v1_search_benchmark.py \
  --budgets 10,25,50,100 --profile-only
```

Do not use MCTS as a distillation teacher until the complete sweep establishes a useful
strength/latency point.

## Reward A/B and replication

Experiments `10` and `11` share seed, runtime, protocol, and training configuration except
the actual/public VP reward treatment. Both final reports must be schema `1.1`, fully
accounted, both-seat, locked-suite evidence with matching protocol signatures. Experiment
`11` is accepted only when its paired outcome gain is at least `0.03` and the R/W/VP gates
do not regress.

After the pair resolves, the queue makes only the winning seed-202 replication ready and
marks the other one skipped. An inconclusive seed-101 comparison unlocks neither replication.

```bash
python examples/colonist_1v1_backlog.py compare \
  10-balanced-actual-s101 \
  11-balanced-visible-s101
```

Development-seed reports are not substitutes for each run's `final_benchmark.json`.

## Prepare legal-CE and listwise BC

Generate one candidate-scored, game-grouped corpus:

```bash
mkdir -p data/hard_state_v2 runs/bc-legal runs/bc-listwise

python examples/colonist_1v1_generate_data.py \
  --num 2000 --teachers F,F --score-candidates \
  --choices-only --seed 101 --output data/hard_state_v2/F_F

python examples/colonist_1v1_generate_data.py \
  --num 2000 --teachers VP,F --score-candidates \
  --choices-only --seed 101 --output data/hard_state_v2/VP_F
```

Train a legal-masked baseline and listwise treatment with the same split and architecture:

```bash
python examples/colonist_1v1_bc.py \
  --data-dir data/hard_state_v2/F_F data/hard_state_v2/VP_F \
  --loss legal_ce --epochs 10 \
  --val-fraction 0.1 --test-fraction 0.1 \
  --split-seed 101 --seed 101 --device auto \
  --out runs/bc-legal/bc.pt --run-dir runs/bc-legal

python examples/colonist_1v1_bc.py \
  --data-dir data/hard_state_v2/F_F data/hard_state_v2/VP_F \
  --loss listwise --hard-states --epochs 10 \
  --val-fraction 0.1 --test-fraction 0.1 \
  --split-seed 101 --seed 101 --device auto \
  --out runs/bc-listwise/bc.pt --run-dir runs/bc-listwise
```

The launcher checks held-out `mean_regret` in both `.meta.json` files and refuses the PPO
warm-start unless the candidate is lower:

```bash
python examples/colonist_1v1_backlog.py start \
  20-hard-bc-actual-s101 \
  --bc-checkpoint "$PWD/runs/bc-listwise/bc.pt" \
  --bc-baseline-checkpoint "$PWD/runs/bc-legal/bc.pt"
```

Experiment `21` uses the same two required flags and is available only when the visible-VP
reward branch was accepted. Lower regret is a preflight requirement, not sufficient gameplay
evidence: the completed run must also improve F rate or F VP margin over its matched parent.

## Promotion and polish

`30-strong-promoted` remains blocked until any complete candidate report has `F >= 10%` and
retains `R >= 90%`, `W >= 70%`, and `VP >= 60%`:

```bash
python examples/colonist_1v1_backlog.py start \
  30-strong-promoted \
  --resume-checkpoint "$PWD/runs/<accepted-parent>/colonist_maskable_ppo.zip"
```

`40-selfplay-polish` requires the accepted strong run and its resume checkpoint. It is kept
only when F/search evidence improves and each weak result stays within two points of the
parent. Self-play is a finishing treatment, not a bypass around the promotion gate.

## Preserve and publish evidence

Before archiving a run, generate a hash-first plan:

```bash
python examples/colonist_1v1_artifacts.py runs/<experiment-id> \
  --keep-latest 3 \
  --pin runs/<experiment-id>/colonist_maskable_ppo.zip
```

Review the JSON, then add `--apply` to move archive candidates under the run's timestamped
`archive/` tree. Nothing is deleted. Publish complete locked evidence separately:

```bash
python examples/colonist_1v1_publish_result.py \
  runs/<experiment-id>/final_benchmark.json \
  --output docs/results/<experiment-id>.json
```

## Stop rules

1. Stop on NaNs, repeated CUDA errors, evaluator errors, incomplete game accounting, a full
   disk, or no progress events for 15 minutes.
2. Do not promote a development report or a report without checkpoint/schema hashes.
3. Do not call a reward winner when the paired gain is below `0.03`; record it as inconclusive.
4. Do not start the 5M profile without `F >= 10%` and all weak gates in complete evidence.
5. Do not infer seat balance from any pre-fix report.
6. Preserve the final checkpoint, schema, manifest, environment lock, evaluation, and registry
   before reversible archival.
